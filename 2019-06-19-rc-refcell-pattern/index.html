为什么 Rc::get_mut 返回了 None？我们看一看文档中关于这个函数的描述：

Returns a mutable reference to the inner value, if there are no other Rc or Weak pointers to the same value.
Returns None otherwise, because it is not safe to mutate a shared value.

也就是说，当且仅当没有其他 Rc 引用内部对象时，Rc::get_mut 才会返回相应的 mutable reference。而在上面的代码中，我们实际上有两个 Rc 引用了内部对象（另一个为 a.obj）。这和 borrow checker 的行为类似，只不过相应的检查被移到了运行时。

从另一个角度看，Rc<T> 和 reference 很类似：

  • 任一时刻允许出现多个 Rc<T>，Rc<T> 支持直接 dereference 得到内部的值。（和 任一时刻可以有多个 immutable reference &T 相比较）
  • 当且仅当只有唯一 Rc<T> 时，Rc::get_mut 才能成功。（和 &mut T 唯一且不能和 &T 共存相比较）。

这其中的问题是，Rc<T> 一旦存在就相当于出现了一个对 T 的 「immutable reference」，但事实上对内部 T 的读取可能只占 Rc<T> 整个生命周期的很小一部分（甚至没有），由此排挤掉 mutable reference 未免过于苛刻了。

但是，我们仍然是需要 Rc 的，因为只有它可以实现共享引用。为解决这个问题，我们要求 Rc<T> 的内部对象 T 可以在不 mutably borrow 的情况下被更改。这就需要 RefCell 的辅助。

RefCell 与 Interior Mutability Pattern

什么是 Interior Mutability Pattern？

Rust 中比较恼人的一点是，可变性具有「传染性」。即若想更改一个对象的一部分，必须持有整个对象的 mutable reference，这对一个有很多互不相关的内部状态的对象来说很不友好。当然很多时候这并不意味着设计的「坏味道」，而是 borrow checker 过于谨慎导致的。为了适应更一般的设计，我们有时需要将对象的可变性隐藏起来，也就是说，在仅持有对象的 immutable reference 的情况下也可以对对象的某一部分进行修改。这就是所谓的 Interior Mutability Pattern。

RefCell 可以实现这一功能。其提供了 .borrow() 和 .borrow_mut() 方法用于借出 immutable 和 mutable reference。这两个函数的参数都是 (&self)，因此不需要 RefCell 本身的（编译时的） mutability。当然，骗过了 borrow checker 并不意味着我们可以突破 Borrowing 的规则——.borrow() 和 .borrow_mut() 在相应情况下还是会 panic，但只要足够谨慎就不会有问题。

Rc + RefCell

我们可以用 Rc<RefCell<T>> 组合出自由度接近 Raw Pointer 的智能指针。一方面 Rc 提供了共享引用的功能；另一方面，RefCell 在必要时才会出借 T 的引用，同时也在运行时检查 Borrowing 规则。这实际上「打破了」Rc 「存在即持有引用」的限制，赋予了程序更多的灵活性。
