深入了解 Flask 的 Thread Local Objects 运行机理

2017年04月27日 | 分类于编程

写在前面

首次接触 Flask 时我就被惊艳到了——Flask 中的请求对象 request 居然是以全局对象的形式提供的！诚然，这种设计避免了繁冗的参数互传，使代码变得简洁，可是在多线程环境中，全局对象真的不会有数据同步问题吗？然而文档明确表明了：这类特殊的全局对象叫 Thread Local Objects，对逻辑而言是全局的，对线程而言则是局部的，因此它们是“线程安全的”（thread-safe)。

心生好奇，遂深究其理，撰此文。

（本文中的代码基于 Flask 0.12，werkzeug 0.12，适当去除了部分 Sphinx 的文档注释）

我们从 request 对象的入口开始探索。request 定义在 flask/globals.py 中——这是一个接口模块，所有的 Thread Local Objects 都由此暴露给调用者。此处只摘取 request 相关的片段：

想必看到这，你仍然心存疑惑——这么两行代码，就能实现上下文环境的动态切换吗？当然，只是实现细节都被很好地封装了起来。不过即使是如此抽象的代码，我们也可以隐约从命名中发掘大致的逻辑：

  • Request 对象外面还包装了一层 Request Context，应该是储存了上下文相关的信息
  • Request Contexts 储存在 _request_ctx_stack 中，这是一个 LocalStack 类型的对象，应该是使用了栈
  • request 和 _request_ctx_stack 之间由一个 LocalProxy 对象相关联，此处应该是使用了代理模式

以上三点可以看出：LocalStack 和 LocalProxy 是两个关键的类。但它们内部是怎样的呢？我们接着往下走。

werkzeug 中的魔法

让我们跳出 flask，进入 werkzeug。werkzeug 是一个 WSGI Utility Library，为 Web 框架的开发提供了诸多便利，也是 pocoo 团队的一个出色的作品。flask 正是基于它构建的。

可以看出，Local 对象将自己的 __storage__ 属性当作一个大字典来使用，并覆盖了 __*attr__ 方法，在增删改属性时加上了一个 ident 标识符——这是一个线程唯一标识符，由 thread.get_ident 获得。如此一来，在不同的线程环境中观察此对象就会得到不同的结果，这就实现了所谓“对线程而言是局部的特性”。
